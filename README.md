# Домашнее задание к занятию "`Базы данных, их типы`" - `Федоров Павел`

### Задание 1 
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?


1.1. `При планировании бюджета проектов, создании аналитических финансовых отчётов и оценке возможных рисков необходимо применять надёжную систему управления базами данных. Такая система должна обеспечивать структурированное хранение информации и её сохранность.
Среди различных типов СУБД особое преимущество имеют реляционные системы. Их главное достоинство заключается в поддержке ACID-свойств, включающих: атомарность операций, согласованность данных, изолированность транзакций, долговечность сохранённой информации.
Кроме того, реляционные СУБД предоставляют широкие возможности для работы со сложными SQL-запросами и позволяют устанавливать различные ограничения для поддержания целостности данных. Это делает их идеальным решением для задач финансового анализа и управления проектами.`

1.2. `Для Лендинга и CRM при условиях что СУБД должна  быть гибкой и быстрой, идеально подходят  NoSQL-СУБД.  
Решение на NoSQL-СУБД обеспецивает: 
-	гибкость схемы данных (например, для хранения неструктурированных или быстро меняющихся данных);
-	масштабируемость (горизонтальное масштабирование для обработки больших объёмов данных и высокой нагрузки);
-	скорость записи данных.`
  
1.3. `Для базы по корпоративным нормам и строго структурированной информации идеально подходят реляционные СУБД они обеспечивают:
-	понятную схему данных;
-	простоту создания отчетов;
-	надежную защиту информации;
-	легкость обучения персонала.`

1.4. `Для формирования маршрутов доставки курьеров лучше всего подходят графовые СУБД.
Графовые СУБД особенно эффективны в случаях, когда требуется анализ сложных взаимосвязей между данными и быстрый поиск связанных объектов. Они становятся все более востребованными в современных проектах благодаря своей гибкости и производительности при работе со связанными данными.`

### Задание 2 Транзакции

 `Этапы транзакции:
1. Начало транзакции
- система получает запрос на пополнение
- открывается сессия работы с базой данных
2. Инициирование операции
- пользователь выбирает способ пополнения (онлайн-банк, терминал, сайт оператора)
- ввод необходимых данных (номер телефона, сумма)
3. Валидация данных
- проверка корректности номера телефона
- проверка допустимости суммы (минимум/максимум)
- проверка достаточности средств на источнике платежа

4. Авторизация платежа
- подтверждение операции пользователем (код из SMS, подпись в чеке)
- проверка статуса карты/счета
- проверка лимитов на операции
5.	Обработка платежа
- блокировка суммы на исходном счете
- проверка возможности зачисления на целевой счет
- обновление балансов
6.	Фиксация результата
- зачисление средств на счет телефона (  если все операции прошли успешно)
- формирование подтверждающих документов
- уведомление пользователя об успешном завершении

7.	Завершение транзакции
- закрывается сессия работы с базой данных`

### Задание 3. SQL vs NoSQL

`3.1 Напишите пять преимуществ SQL-систем по отношению к NoSQL.`

1. `Стандартизированный язык запросов. Стандартизированный, декларативный язык, который легко изучать и использовать даже новичкам, при этом он одинаково работает на разных платформах.`
2. `Поддержка транзакций. (принцип ACID: атомарность, согласованность, изолированность, долговечность). Это гарантирует целостность данных даже в случае системных сбоев.`
3. `Безопасность данных. Реляционные СУБД поддерживают шифрование на уровне таблиц или полей.`
4. `Широкие возможности для аналитики. Благодаря реляционной алгебре можно комбинировать данные из разных таблиц, агрегировать значения и строить сложные запросы без изменения структуры базы.`
5. `Строгая целостность данных. В SQL-системах можно устанавливать ограничения на значения, которые записываются в таблицы, и гарантировать их соблюдение до и после транзакции.`


### Задание 4 . Кластеры

`Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.

На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?`

Ответ:
1. Критерии для выбора СУБД:
1.	Тип нагрузки (OLTP vs OLAP). Нужно определить, требуется ли обработка транзакций в реальном времени (OLTP) или аналитическая обработка больших объёмов данных (OLAP).

2.	Объём данных и требования к масштабируемости. СУБД должна поддерживать горизонтальное масштабирование (добавление узлов в кластер) и эффективно распределять нагрузку между ними.

3.	Тип данных и требования к обработке. Если данные структурированные и требуется поддержка SQL, подойдут реляционные или колоночные СУБД (например, Greenplum, ClickHouse). Для неструктурированных или полуструктурированных данных могут быть уместны NoSQL-системы. Если данные имеют сложные взаимосвязи, стоит рассмотреть графовые СУБД.

4.	Производительность. Важно оценить скорость обработки запросов. ClickHouse специализируется на высокоскоростной аналитике и векторизованном выполнении запросов, что значительно ускоряет обработку. Greenplum оптимизирован для аналитических запросов, но может уступать в скорости при частом обновлении данных.


5.	Совместимость с экосистемой и инструментами. СУБД должна интегрироваться с существующими инструментами и рабочими процессами.

6.	Надёжность и отказоустойчивость. Система должна обеспечивать репликацию данных и восстановление после сбоев.

7.	Стоимость и ресурсы. Нужно учитывать затраты на инфраструктуру, лицензирование, обслуживание и администрирование.


Для обработки данных на 1000 машинах подойдет MapReduce.
Модель, при которой данные разделяются на части, обрабатываются параллельно на разных узлах (этап Map), а затем результаты агрегируются (этап Reduce). Данная модель идеально подходит для пакетной обработки больших объёмов данных с акцентом на экономичность и отказоустойчивость.
